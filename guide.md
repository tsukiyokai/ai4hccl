## 这份指南要解决什么问题

conv算子组用AI辅助开发，最终代码70%以上由AI生成。但HCCL不能照搬——两者的复杂性结构完全不同。ops的代码高度模板化，AI一抄一个准；HCCL的核心是并发控制和通信协议，AI在这方面基本帮不上忙。我们挖了repo-dig的缺陷数据、复盘了conv算子组的实战经验，最后得出的结论是：HCCL综合提效15-20%，远低于ops的40-50%。

这份指南给出27条策略，覆盖每个环节AI能做什么、不能做什么、怎么做。

核心思路：把有限的AI投入砸在偶然复杂性最集中的地方。

## HCCL到底难在哪

### 背景

HCCL是昇腾AI处理器的集合通信基础设施，支持单机多卡到32K卡分布式训练。代码规模：HCCL主仓约65K行，HCOMM底层库约55万行，13种集合通信算子，多级通信拓扑，多种通信算法自适应选择。

### 为什么不能照搬ops的经验

先看数据。

表1 HCCL与周边算子组复杂性对比

| 维度         | 周边算子组(ops)                            | HCCL通信库                         | 对AI策略的影响                                      |
| ------------ | ------------------------------------------ | ---------------------------------- | --------------------------------------------------- |
| 代码模式     | 算子间高度相似（tiling/kernel/host三件套） | 13种通信算法逻辑各自独特           | ops适合模板仿写（实测最大提效点），HCCL仿写空间有限 |
| 核心困难     | 精度、NPU硬件约束、向量化                  | 并发正确性、分布式一致性、协议兼容 | HCCL的核心困难更难被AI消除                          |
| 样板代码占比 | 高（CMake/host/tiling模板）                | 低（逻辑代码占比高）               | HCCL中AI可消除的偶然复杂性比例更小                  |
| 代码规模特征 | 单算子规模小（数百行kernel）               | 单文件规模大（God Object达9152行） | HCCL的AI辅助受上下文窗口限制更为严重                |
| 缺陷可审查性 | 95%（模式化缺陷为主）                      | 85%（并发类缺陷隐蔽）              | HCCL需补充并发专项审查，AI覆盖率上限更低            |
| 测试特点     | 单卡UT/精度对比                            | 多卡集合通信（需物理设备）         | HCCL测试环境搭建的自动化价值更高                    |

说白了：ops的代码像流水线上的标准件——AI照着模板抄就行，抄完70%的代码就出来了。HCCL的代码像手工打造的精密仪器——13种通信算法每种都不一样，并发控制逻辑让AI写，大概率埋几个竞态条件进去。

关键数字：ops的偶然复杂性约60%，HCCL加权约30%。差了一倍。这个差距决定了AI在HCCL中的天花板。

### 本质复杂性vs偶然复杂性

Brooks在1986年写过一篇著名的文章叫"No Silver Bullet"，把软件的困难分成两种：

- 本质复杂性：问题本身就这么难，换什么工具都消除不了。HCCL里就是并发控制、通信算法、OpCode协议兼容这些东西。
- 偶然复杂性：工具和流程带来的额外摩擦。CMake脚本维护、测试脚本编写、日志格式化，这些换个好工具就能省掉。

AI消除的是偶然复杂性。高级语言替代汇编、分时系统替代批处理，历史上每次大的生产力提升都是在消除偶然复杂性。AI不是银弹，但它是一把好用的螺丝刀。

我们评估了HCCL各环节的偶然复杂性占比：

表2 HCCL各环节偶然复杂性占比

| 环节     | 偶然占比 | AI提效空间 |
| -------- | -------- | ---------- |
| 构建     | ~70%     | 大         |
| 测试     | ~40%     | 中         |
| 编码     | ~35%     | 中         |
| 上板验证 | ~30%     | 小-中      |
| 代码检视 | ~15%     | 小-中      |
| 需求分析 | ~10%     | 小         |
| 详细设计 | ~10%     | 小         |
| 问题定位 | ~10%     | 小         |

构建环节70%是偶然复杂性——CMake脚本漏文件、链接依赖缺失这种事，写个脚本就能查。问题定位只有10%——让AI定位一个多机死锁？算了吧。

### 八条基本原则

在HCCL这种本质复杂性占70%的系统里，"知道AI不该做什么"比"知道AI能做什么"重要得多。以下八条贯穿所有环节：

C1 偶然复杂性低于20%的环节，AI只做信息整理，不参与决策

- 适用：需求(10%)、设计(10%)、问题定位(10%)
- 怎么判断越界了：AI的输出被当成技术决策依据，而不是参考材料

C2 凡涉及正确性判断，必须有确定性验证手段

- 编译器、TSan、ASan、测试——这些说了算
- "AI审查通过了所以并发逻辑没问题"——这话谁说谁负责

C3 变更影响面越大，AI自治度越低

- 跨层/跨版本/跨设备变更 → 人工主导
- 格式修正/文档更新 → 全自动，放心让AI干

C4 AI漏掉的问题必须沉淀下来

- 能写成规则的入规则库，不能规则化的入checklist
- 同一类问题被AI反复漏掉但没人更新规则 → 流程出了问题

C5 喂给AI的上下文：200-500行精准相关代码

- God Object（9152行）禁止整体输入——把整个communicator文件粘进去，AI只会生成更多垃圾
- 上下文太多反而降低生成质量，这不是直觉，是实测结论

C6 每次AI生成代码不超过50行

- 生成后未经编译验证的代码是负债，不是资产
- 一次性让AI写几百行？恭喜，调试这些代码花的时间比手写还多

C7 环节衔接：每个环节的产出必须是AI可消费的格式

- 需求 →（md+PlantUML+接口定义+参考资料）→ 设计
- 设计 →（md+PlantUML源码+正常/异常分支全覆盖）→ 编码
- 编码 →（代码+UT+AI检视报告）→ 检视
- 上游只出了doc/docx或纯截图？那下游AI就是个瞎子

C8 AI没有合入权限

- 不给AI +1/+2权限。AI的所有输出必须经人工确认后才有流程效力
- "AI检视通过了"不等于"检视通过了"

## 逐环节AI辅助策略

### 需求分析

责任方：SE。输入：论文、竞品分析、客户需求。输出：需求文档。

需求分析几乎全是本质复杂性——MPI语义适配、硬件拓扑约束、性能指标推导、NCCL/RCCL竞品分析，全部需要领域专家判断。AI在这里就是个搜索引擎加整理员。

判断标准：AI做的事，换一个不懂通信的实习生加搜索引擎也能做 → 用对了。需要领域判断才能完成 → 越界了。

前置条件：无特殊前置，研发链起点。

红线：

1. AI不做技术判断——MPI语义、拓扑约束、性能指标、竞品选型，全部SE来
2. AI的竞品对比/论文摘要仅作参考，不是决策依据
3. 并发安全不变量（"不得死锁"）必须人工定义

产出格式(C7)：md格式+PlantUML图+接口定义+参考资料。

#### 场景1: 论文/竞品分析辅助

AI读论文、整理竞品对比表。拿到整理好的材料后做判断。

对比维度要包括：通信算法（Ring/RHD/NHR等）、拓扑适配(HCCS/PCIe/RoCE)、算子覆盖度。

#### 场景2: 需求结构化模板

用EARS格式(Event-Action-Response-State)把通信需求结构化，直接映射为测试用例。

示例："When [rank count > 8 and topology = mesh], the system shall [select NHR algorithm] within [100ms setup time]"。

HCCL特有要求：并发安全不变量（如"任何通信算子在任意拓扑下不得死锁"）必须作为需求文档的必填项，不能留到设计阶段。ops不需要这步，我们必须有。

### 详细设计

责任方：MDE。输入：需求文档。输出：详细设计文档。

设计工作也几乎全是本质复杂性。13种通信算法各自独特、并发模型设计、状态机设计、OpCode协议兼容——每份设计文档都不一样。ops的设计以"tiling策略+kernel方案"为核心，相对模板化，AI准确率能到80%。HCCL的设计独特性高，AI准确率预估50-60%。

vibe-design当前适配度：0%。没有HCCL专项模板（并发规格、协议兼容、状态机、性能设计），需要从头建。

前置条件：

1. 需求文档已以md格式交付且包含接口定义
2. 并发安全不变量已明确

红线：

1. AI生成的方案不能直接用——只是思考起点
2. 每个AI方案须回答三个问题：并发安全怎么保证？和现有OpCode兼容吗？God Object要改吗？答不清楚就只当brainstorm素材
3. 并发设计规格（锁粒度、同步原语、屏障位置）不由AI决定

产出格式(C7)：md格式+PlantUML源码+正常/异常分支100%覆盖。STC评审必须严格把关分支覆盖——这些分支是编码阶段AI的判断基准，分支没覆盖全 → AI生成的代码必然有遗漏。

判断标准：AI只做"搬砖"（文档骨架、图表绘制、格式转换），不做"决策"（算法选择、并发模型、协议设计）。

#### 场景1: 设计文档结构化辅助

AI根据讨论纪要生成文档骨架（章节结构+占位描述），人工填核心决策。

HCCL模板必须包含这些特有章节：

- 并发设计规格：锁粒度、同步原语选择、顺序约束、屏障位置
- 通信协议兼容性：OpCode版本策略、新旧节点互操作方案
- 状态转换图：executor生命周期(sole/parallel/concurrent/sequence)
- 性能设计：内存对齐、cache策略、pipeline配置

#### 场景2: 图表自动生成

AI根据文字描述生成PlantUML图（时序图、状态机图、组件图）。把白板照片/文字描述转成规范PlantUML，这个AI很擅长。

并发时序图需人工校验——AI可能漏掉关键同步点。

#### 场景3: Disposable Bets

让AI扮演不同角色（并发审计员/协议审查员/性能工程师）分别产出2-3个方案，从中挑选。

这些方案是一次性的(disposable)，用完就扔。HCCL设计的本质复杂性需要人脑判断。

文档格式要求（从ops复盘迁移的教训）：

- Markdown格式（doc/docx AI理解效果差）
- 图表用PlantUML/Mermaid文本，禁止纯截图
- 按模块组织，每个模块对应一个独立的代码生成任务
- 精度标准/性能基线/测试方案在设计阶段就定死（ops教训：没左移导致返工10天）

### 编码

责任方：开发人员。输入：详细设计文档。输出：功能代码+UT+LLT。

编码是本质和偶然并存的环节。偶然部分(35%)：目录骨架、设备分支、API注册、错误处理模板、日志打印。本质部分(65%)：并发控制、通信算法实现、OpCode协商、状态机逻辑。

AI代码占比预估：30-40%（vs ops的70%）。差距来源：HCCL可以给AI写的偶然代码比例更小。

有四个区域AI绝对不能碰：

表4 AI编码禁区

| 禁区           | 原因                               | 缺陷证据                                        |
| -------------- | ---------------------------------- | ----------------------------------------------- |
| 并发控制逻辑   | AI并发推理能力弱；状态空间指数增长 | HCCL并发缺陷10条/11.9%（全P0），HCOMM 11条/6.8% |
| 通信算法实现   | 13种算法各自独特，非模板可仿写     | 算法正确性缺陷21条/25.0%（HCCL最大缺陷类）      |
| OpCode版本协商 | 协议兼容性属P0级本质约束           | 配置与兼容性缺陷13条/15.5%                      |
| 状态机转换逻辑 | 需对executor生命周期深度理解       | aicpu_communicator.cc中192行状态机while循环     |

这不是我们保守。并发缺陷全是P0，10条里每一条都能搞出生产事故。让AI写并发控制代码，就像让实习生设计核电站的安全系统。

前置条件：

1. 设计文档已完成并review通过（含STC评审的分支100%覆盖）——不跳过设计直接让AI编码
2. God Object涉及的3个热点文件，按功能分区准备200-500行精准上下文+接口签名，禁止整体输入
3. 错误处理模式库已建立——AI引用标准模式，不自己发明
4. 远程编译环境可用——每次生成后须即时编译验证

红线：

1. 四个禁区（硬性）：并发控制/通信算法/OpCode协商/状态机——AI不写实现（可以写接口签名和注释占位）
2. 每次AI生成不超过50行；生成后：编译通过+UT通过+人工review并发相关代码
3. 并发代码变更须附带TSan验证结果——无TSan证据的并发修改不得合入
4. AI不自创错误处理模式——新模式需code review后入库
5. AI不判断哪些错误该重试、哪些该立即失败——这是业务语义

TDD约束：

1. 先让AI写UT，UT人工确认后冻结，后面禁止AI再改UT——UT是正确性的锚点，不能随实现代码漂移
2. AI生成的UT必须人工检视——AI可能生成形式上通过但实质不验证任何东西的UT
3. AI生成的Checker必须人工检视

PR前约束：

4. PR前必须用AI检视，并确认AI意见
5. PR前除了AI检视还必须人工本地检视——AI检视不替代人工检视，两者都是必要条件

判断标准：AI能写的代码 = 换一个不懂并发的程序员照模板也能写出来的代码。需要理解线程交互、分布式一致性、协议演进的代码 = 禁区。

#### 场景1: 算子骨架生成器

输入新算子名称+通信语义描述+参考算子名称，输出ops/目录下的algo/executor/selector/template四件套骨架。

prompt模板："参考HCCL中[AllReduce]算子的目录结构，为新算子[X]生成骨架代码。只生成文件结构、类声明、接口签名，不实现通信逻辑和并发控制。"

只生结构和接口，不生并发和通信逻辑。生成后须编译验证。

#### 场景2: 小粒度迭代编码规范

从ops复盘迁移并加强：

- 每次AI生成不超过50行
- 每次生成后须：(1)编译通过；(2)相关UT通过；(3)人工review并发相关代码
- 并发代码变更须附TSan验证结果

ops教训直接适用："一次性生成tiling+kernel全部代码，冗余多、幻觉多，导致大量返工"。HCCL返工风险更高——ops返工主因是精度问题（可测试验证），HCCL返工主因是并发问题（跑100遍测试可能都复现不了）。

#### 场景3: God Object上下文管理

三个热点文件占全部缺陷的36.9%：

- hccl_communicator_host.cc: 9152行
- aicpu_communicator.cc: 5550行
- communicator_impl.cc: 3789行

别把这些文件整个喂给AI。管理策略：

- 按功能分区，每次只给200-500行+接口签名
- proximity优先：以同算子的algo/executor/selector/template作为上下文
- 渐进式摘要：对God Object每个方法生成一句话摘要，当"方法索引"用
- 长期方案：按功能内聚逐步拆分（人工主导）

#### 场景4: 错误处理模式库

把HCCL标准错误处理模式沉淀成code snippet库，AI生成代码时自动引用。

HCCL特有模式：

- ACL接口返回值检查
- 通信超时处理
- 资源初始化失败的回滚（RES-01~05规则）
- 日志tag匹配（LOG-01~04规则：格式串参数对应、tag匹配、快速路径避免日志洪泛）

AI用已有模式，不自创新模式。新模式需code review后入库。不由AI决定哪些错误该重试、哪些该失败——这是通信库的业务语义。

#### 场景5: 远程开发模式优化

本地IDE(Claude Code/Cursor) + 远端编译验证。

- AI在本地生成代码，rsync/ssh同步到远端编译
- 编译结果自动反馈回IDE，形成闭环
- AI工具在本地跑（延迟低），只有编译走远端

ops复盘经验："本地承担'思考和写代码'的角色，远端承担'编译和跑性能'的角色"。HCCL编译依赖链比ops更深(ACL→HCCL→HCOMM→Runtime→Driver)，还需交叉编译，编译反馈周期更长，减少不必要的编译往返尤为重要。

### 构建

责任方：开发者+CI。输入：代码。输出：编译产物。

构建是偶然复杂性最集中的环节，也是AI投入ROI最高的环节。CMake脚本漏文件、条件编译分支遗漏、链接依赖缺失——这些占了大量的构建缺陷。数据支撑：HCOMM 162条缺陷中24条(14.8%)是构建/编译/链接类（频次最高），HCCL-dev 10条中3条(30%)是构建/链接类。

前置条件：

1. B-1（CMake守卫）和B-2（编译器诊断全开）已部署——这两项是ROI最高的投入，在引入任何LLM辅助工具之前必须先落地
2. 构建矩阵文档（设备类型x编译选项x依赖版本）已建立

红线：

1. AI的构建修复方案必须通过完整编译验证——不接受"注释掉报错代码"
2. AI不动交叉编译目标选择和依赖版本锁定——业务约束
3. 编译器诊断渐进式开启——首次全开可能几百个warning，分批修，不让AI一次"修完"

判断标准：构建环节AI是"守门员"（拦截问题），不是"创造者"（生成代码）。CMake守卫和编译器诊断比任何LLM方案都便宜且可靠。

#### 场景1: CMake一致性守卫

在CI中加个脚本：`find src/ -name "*.cc" | sort` vs `grep "\.cc" CMakeLists.txt | sort`，差集就是漏掉的源文件。

一个shell脚本，投入低到几乎为零，预期拦截40-50%构建类缺陷。HCCL的CMake比ops更碎片化（ops新增算子有固定CMake模式），一致性检查价值更大。

#### 场景2: 编译器诊断全开

配置：`-Werror -Wshadow -Wconversion -Wformat -Wsign-compare`

预期拦截约15%总缺陷。投入也是极低（改个编译选项）。注意首次开启warning可能很多，渐进式修复。

B-1+B-2组合效果：B-1拦截40-50%构建类缺陷（CMake遗漏/链接依赖缺失），B-2额外拦截约15%跨类型编码缺陷（变量遮蔽/类型转换/格式串错误），投入极低。这是HCCL全部AI辅助策略中ROI最高的组合。

#### 场景3: AI辅助构建错误修复

遇到编译/链接错误，把错误日志给AI分析。

prompt模板："HCCL构建报错如下：[错误日志]。项目使用CMake构建，依赖ACL/HCOMM/Runtime。请分析根因并给出修复方案。"

AI给的修复必须过完整编译验证。参考数据：CI repair成功率17%（GPT-3.5, 170K行上下文），但HCCL构建上下文远小于170K行，实际成功率应该更高。

#### 场景4: 构建矩阵文档化

让AI辅助维护构建矩阵文档（设备类型x编译选项x依赖版本）。HCCL需同时支持Ascend 910/310/710和多种主机架构，编译矩阵比ops复杂得多。

### 代码检视

责任方：开发人员（多人）。输入：代码。输出：修正后代码。

代码检视是本质复杂性最集中的环节之一。85%的复杂性来自需要reviewer深度理解的领域：并发逻辑审查、跨层影响分析、协议兼容性验证、缓存一致性判断。

vibe-review适配度：90%。80条HCCL项目级规则已内置，覆盖算法正确性(ALG-01~08)、并发(CON-01~09)、资源生命周期(RES-01~05)、日志(LOG-01~04)等。10%缺口在误报排除和跨层影响分析。

HCCL可审查性：85%（vs ops的95%）。10%差距来自并发类缺陷的隐蔽性——竞态条件需要特定线程调度才暴露。提升路径(85%→90%)：不靠更多AI规则，靠TSan/ASan覆盖率从0→80%。

前置条件：

1. vibe-review已配置HCCL项目级规则（80条）
2. P0变更（涉及并发控制）：TSan/ASan基础设施可用

红线：

1. AI检视结论不能替代TSan/ASan验证——AI说"并发安全"不算数，TSan说"无竞态"才算数
2. AI不做并发逻辑的最终正确性判断——Layer 3必须人工
3. AI标注的可疑并发问题，即使人工觉得没事，也要记录理由——防止盲目自信

责任链：

1. AI检视结果必须由开发人员确认；committer +1时必须确认开发人员的反馈
2. committer对最后一版AI检视报告负责——确认残留的AI检视意见都是误报或已知遗留
3. 发现AI误报必须反馈@fanshanshan → 规则库更新 → 下一轮检视更准
4. AI没有+1/+2权限(C8)

判断标准：AI检视的成功标准不是"发现了多少bug"，而是"释放了reviewer多少注意力给Layer 3"。reviewer还在看格式和规范问题 → Layer 1没达标。

#### 场景1: 分层检视流程

- Layer 1（全自动）：vibe-review跑编码规范+已知缺陷模式。预期节省reviewer 30-40%时间。
- Layer 2（AI辅助+人工确认）：AI标注可疑的并发/缓存/兼容性问题，人工判断。
- Layer 3（人工主导）：并发逻辑正确性、跨层影响分析。AI不参与。

流程：代码提交 → AI自动跑Layer 1 → 并行启动Layer 2和人工检视 → 人工聚焦Layer 3 → 修复全部意见后关闭。

#### 场景2: Delta-first Review

AI先分析本次变更影响哪些通信路径，生成影响摘要。reviewer从影响摘要入手，不逐行看diff。

prompt模板："分析以下HCCL代码变更的影响范围。列出：(1)直接修改的函数；(2)调用链上游；(3)调用链下游；(4)是否涉及并发控制、缓存、协议兼容性。"

HCCL的局部修改可能破坏跨层通信语义——改个executor → 影响selector策略 → 影响API语义。这个问题ops没有，我们必须专门应对。

#### 场景3: N-version并发审查（P0变更专用）

涉及并发控制的P0级变更，让2个独立AI（不同prompt/不同上下文）分别审查，交叉对比，不一致处标高风险。配合TSan/ASan验证。

并发安全需要确定性证据（TSan输出），不能仅靠概率性意见（AI判断）。这是ops不需要但HCCL必须做的步骤。

#### 场景4: 审查规则持续演化

每次人工review发现AI漏掉的缺陷：

- 能规则化 → 更新vibe-review的HCCL规则库
- 不能规则化 → 记入"需人工关注"的checklist

把审查经验编码为持久化规则，不靠AI的"记忆"。

### 开发者测试

责任方：开发者。输入：功能代码。输出：UT+LLT(Checker)+测试报告。

本质复杂性来自分布式通信的不可约简性——多卡多机测试需物理设备（设备本身就是瓶颈）、测试设计需理解通信语义、并发场景确定性复现困难（LLM故障复现率0%）。偶然部分(40%)集中在测试脚本编写（hccl_test参数组合）、环境配置脚本、测试框架代码。

前置条件：

1. 性能基线已在设计阶段确定——否则测试结果分析无基线可比
2. 测试环境（多卡/多机）可获取
3. 测试规范要求（功能、性能、长稳、整网）已明确

红线：

1. 必须基于测试规范生成用例——AI不得自行发明测试类别
2. AI不生成并发测试的时序断言
3. AI不判断通信结果正确性（如AllReduce结果是否等于各rank输入之和）
4. AI生成的UT桩代码必须编译验证——LLM对私有API编译率预估<30%
5. AI生成的参数组合需人工确认覆盖度——AI可能漏掉边界条件（消息大小为0、卡数为1）

判断标准：AI做"体力活"（参数排列组合、环境配置模板、日志解析），不做"脑力活"（验证逻辑、正确性断言、并发场景构造）。

#### 场景1: hccl_test矩阵生成器

输入算子列表+参数范围，输出完整测试命令矩阵+执行脚本。

prompt模板："为以下HCCL算子生成测试矩阵：[AllReduce, AllGather, ReduceScatter]。参数范围：卡数[2,4,8]，数据类型[fp16,fp32]，消息大小[1KB,1MB,1GB]。输出bash脚本，每个组合一行hccl_test命令。"

测试脚本编写时间-80%。这是ops"分钟级完成（人工需数小时）"的最大提效点之一，直接复用。

#### 场景2: 多机环境配置模板

AI生成rank_table.json模板+环境变量设置脚本+SSH免密配置checklist。覆盖：单机多卡、双机、四机全互联。

ops上板是单卡验证，HCCL需要多卡多机协同，环境配置复杂度高出一个数量级（网络拓扑、HCCL_SOCKET_IFNAME、RDMA参数）。这是HCCL中AI辅助测试的独特价值点。

#### 场景3: UT桩代码生成（受限使用）

AI生成测试框架代码（setUp/tearDown、mock对象），人工填通信逻辑验证。

限制：不由AI生成并发测试的时序断言、不由AI判断通信结果正确性、私有API编译率预估<30%。

ops的UT是单卡确定性输入输出，HCCL UT需模拟多rank交互，AI生成难度更高。

#### 场景4: 测试结果自动分析

AI写脚本解析hccl_test输出，提取带宽/延迟数据，与历史基线对比，标注性能退化。

HCCL的性能指标与通信拓扑强相关（HCCS vs PCIe vs RoCE性能差异可达10倍），AI可自动关联拓扑配置与性能数据。ops的性能基线与硬件拓扑无关。

### 上板验证

责任方：开发者。输入：编译产物+测试脚本。输出：验证结果。

本质困难：硬件交互不可替代——物理设备获取（多卡多机是稀缺资源）、NPU行为不可预测（温度/功耗/硬件故障）、端到端集成、性能调优。偶然部分(30%)集中在环境配置、配置文件准备、测试执行脚本、日志收集整理。

前置条件：

1. 环境检查清单已建立（NPU型号/SDK版本/镜像版本/网络配置/RDMA参数）
2. 编译环境与运行环境已确认一致——ops教训：联调环境信息未提前对齐

红线：

1. AI对上板日志的分析仅做初步分类（环境问题vs功能问题）——功能问题根因必须人工分析
2. AI不直接改HCCL运行时参数来"修复"上板失败——可能掩盖真正问题
3. 多机日志的因果推理不由AI做——AI可以重建时间线，人工做因果分析

判断标准：AI做"准备"（环境检查、配置生成）和"整理"（日志收集、时间线对齐），不做"诊断"（根因分析、参数调优）。

#### 场景1: 环境准备自动化

AI生成环境检查脚本：检查NPU型号、driver版本、SDK版本、网络连通性。一键执行，输出"就绪"或"以下不满足"。

ops教训："联调环境信息未提前对齐，SDK版本不匹配、编译包跨平台不可用"。HCCL环境复杂度更高（多机网络拓扑、RDMA参数），自动化检查价值更大。

#### 场景2: 上板日志分析辅助

上板失败时，AI初步分析日志：区分"环境问题"（配置/版本/网络）与"功能问题"（代码bug）。环境问题AI可以给修复建议，功能问题交人工。

HCCL的上板日志涉及多rank多机的交叉信息，AI可辅助按rank/timestamp关联日志，构建通信时间线。

联调环境对齐检查清单（从ops迁移）：

- 编译环境（毕昇编译器版本、CMake版本）
- 芯片型号(Ascend 910/310/710)
- SDK版本（cann-toolkit版本）
- 网络配置（HCCL_SOCKET_IFNAME、RDMA参数）
- 镜像版本

### 问题定位

责任方：开发者/问题接口人。输入：问题描述+日志。输出：原因+修正方案。

问题定位是AI最难介入的环节。实话说，让AI找一个Heisenbug（通信超时、死锁那种偶尔出现又无法复现的）？研究数据显示LLM故障复现率0%。Bug localization MAP只有0.39——意味着AI指的方向约60%是错的。

前置条件：

1. 问题知识库(P-1)已建初版
2. 多机日志已收集且时间戳已对齐

红线：

1. AI给的根因候选（3-5个文件/函数）仅作参考——MAP 0.39意味着大概率是错的
2. AI不做分布式因果推理——多rank间消息因果关系需人工判断
3. AI不给"问题已定位"的最终结论——只给候选范围
4. Heisenbug不由AI定位——触发条件含硬件状态/网络抖动/线程调度等不可控因素
5. AI定位无效时，本次case必须补充到问题知识库——闭环

判断标准：AI做"缩小范围"（从430个源文件到3-5个）和"模式匹配"（知识库检索），不做"推理"（因果分析、Heisenbug复现）。

#### 场景1: 问题知识库

把历史问题积累成结构化知识库：{错误码，日志特征，根因类别，修复方案}。新问题进来先检索匹配已知模式。

通信库的问题模式相对集中——超时、死锁、性能退化、协议兼容性错误。知识库围绕这些模式设计。

#### 场景2: 多机日志分析辅助

AI写日志解析工具：收集多机日志 → 时间戳对齐 → 按rank构建通信时间线 → 标注异常事件（超时/错误/重试）。人工从整理好的时间线中分析根因。

ops的问题多数DumpTensor+精度对比就能单卡复现。HCCL的问题需要多机日志关联分析，这是我们独有的需求。

#### 场景3: 候选范围缩小

AI根据错误日志和git log，缩小可能的根因范围到3-5个文件/函数。人工从缩小后的范围精确定位。

流程：AI做问题定位（知识库匹配+日志分析+候选缩小）→ 人工确认。有效就修；无效就把本次case补充到知识库。

### Skills开发实践

Skills是Claude Code的能力扩展机制——一个SKILL.md文件就能为AI注入领域知识和专项工作流。vibe-review（R-1分层检视）和vibe-design（D-1设计模板）都以Skill形式实现。

Skill目录结构：

```
<skill-name>/
├── SKILL.md       # 核心：frontmatter + 指令正文（必需）
├── references/    # 参考资料（规则库、大型文档，按需加载）
├── scripts/       # 辅助脚本
└── docs/          # 补充文档
```

SKILL.md编写规范：

| 要素         | 规范                                                    |
| ------------ | ------------------------------------------------------- |
| frontmatter  | `name` + `description`（含触发词）+ 可选`allowed-tools` |
| description  | 列举触发短语，确保用户自然表达即可触发                  |
| 正文语气     | 祈使句为主（"分析…"、"生成…"、"检查…"）                 |
| 篇幅         | 1500-2000字——太短信息不足，太长稀释注意力               |
| 参考资料引用 | 正文中用Read指令按需加载references/下的文件             |

渐进式信息公开——Skill对AI暴露信息分三层，避免一次性灌入过多上下文（C5原则的工具级体现）：

1. metadata：frontmatter中的name+description → Claude Code据此决定是否触发
2. SKILL.md正文：核心流程和约束 → 触发后加载
3. references/+scripts/：详细规则库和辅助脚本 → 正文中按需加载

验证清单：

- [ ] description包含用户常用的触发短语
- [ ] 正文不超过2000字
- [ ] 大型参考资料放在references/下，正文中按需加载
- [ ] 本地测试：在Claude Code中用自然语言触发，验证是否命中

## 附录

### AgenticSE四大悖论与HCCL缓解策略

这四个悖论来自AgenticSE研究，听着学术，但每个都在HCCL的实践中真实存在。用大白话说：

表14 AgenticSE悖论在HCCL中的表现及缓解

| 悖论              | 大白话                                   | 在HCCL中的表现                     | 我们怎么应对                              |
| ----------------- | ---------------------------------------- | ---------------------------------- | ----------------------------------------- |
| Eagerness Paradox | AI越快越容易跑偏——它不会停下来想"我对吗" | 快速生成的并发代码可能引入竞态条件 | C-2小粒度迭代 + Beyond Done（TSan验证）   |
| Context Paradox   | 给太多上下文反而更糟——AI被信息淹没了     | 65K LOC不能全部输入AI              | C-3 God Object上下文管理（分区+方法索引） |
| Tunnel Vision     | 局部完美但破坏全局——AI只看眼前这片代码   | 改executor可能破坏跨层通信语义     | R-2 Delta-first Review（影响范围分析）    |
| Learning Paradox  | AI不积累经验——每次都从零开始             | 并发bug pattern需要持久化          | R-4审查规则演化 + P-1问题知识库           |

Eagerness Paradox是最危险的。AI写了一段并发代码，信誓旦旦说"线程安全"——一跑TSan，竞态条件满天飞。AI的问题不是不够聪明，而是太自信。所以我们要求：每次并发代码变更必须附TSan结果，AI说安全不算数。

Context Paradox在God Object上体现得淋漓尽致。communicator_host.cc有9152行，全量输入AI后生成质量反而下降。切200-500行精准上下文，效果更好。违反直觉，但这是实测结论。

Tunnel Vision在HCCL特别致命。ops里改一个kernel，影响范围就那个kernel。HCCL里改一个executor的逻辑，可能影响selector的策略选择，进而影响API的语义。局部看没问题，全局看出大事了。所以P0变更必须做Delta-first Review。

Learning Paradox意味着不能指望AI"学会"上次的经验。每次人工review发现AI漏掉的缺陷，必须编码成持久化规则（vibe-review规则库或checklist），否则下次AI还是会漏。

### HCCL缺陷模式速查

数据来源：repo-dig深度分析（HCCL 84条缺陷/428次提交，HCOMM 162条缺陷/3084源文件）。

表15 HCCL缺陷分类及AI可检测度

| 缺陷类别     | HCCL频次        | HCOMM频次 | 严重度 | AI可检测度                       |
| ------------ | --------------- | --------- | ------ | -------------------------------- |
| 算法正确性   | 21(25.0%)       | —         | P0-P1  | 中（单线程逻辑可检测，并发不可） |
| 配置与兼容性 | 13(15.5%)       | —         | P0-P1  | 中高（规则化检测）               |
| 并发问题     | 10(11.9%)       | 11(6.8%)  | P0     | 低（40%可规则化，20%需运行时）   |
| 日志与调试   | 8(9.5%)         | —         | P2     | 高（格式串匹配等确定性检查）     |
| 资源生命周期 | 8(9.5%)         | —         | P1     | 中高（模式化检测）               |
| 错误处理     | 7(8.3%)         | —         | P1     | 高（返回值检查等模式明确）       |
| 缓存一致性   | 6(7.1%)         | —         | P1     | 低（需理解业务语义）             |
| 构建/链接    | 3(hccl-dev 30%) | 24(14.8%) | P2     | 高（CMake检查+编译器诊断）       |

几个值得注意的点：

- 算法正确性缺陷占25%，是HCCL最大的缺陷类。但其中涉及并发的部分AI检测不了。
- 并发问题只占11.9%的频次，但严重度全是P0。10条里每一条都可能是生产事故。
- 构建/链接类在HCOMM中占14.8%（频次最高），大部分可以被B-1和B-2拦截——这就是为什么我们说构建是ROI最高的环节。
- 日志和错误处理类AI检测度最高，这正是vibe-review Layer 1的覆盖重点。
