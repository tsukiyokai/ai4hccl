## 摘要

本指导书针对HCCL集合通信库的业务特点，系统分析了AI辅助研发在8个环节（需求分析、详细设计、编码、构建、代码检视、开发者测试、上板验证、问题定位）中的适用性与边界。基于Brooks本质/偶然复杂性理论框架[1]，结合repo-dig缺陷挖掘的实证数据[5]和conv算子组的实战复盘经验[6]，本文提出27条分级AI辅助策略(Tier 1-4)，并给出三阶段实施路线图。

核心结论：HCCL各环节偶然复杂性加权占比约30%（远低于周边算子组的60%），综合提效预估15-20%。AI投入应采取精准投放策略，优先覆盖构建（偶然占比70%）和测试（偶然占比40%）环节，严格限制在并发控制、通信算法、协议设计等本质复杂性领域的应用。

## 总论

### 背景与目标

HCCL是昇腾AI处理器生态的集合通信基础设施，支持单机多卡到32K卡的分布式训练。代码规模约65K行（HCCL主仓） + 55万行（HCOMM底层库），涉及13种集合通信算子、多级通信拓扑、多种通信算法的自适应选择。

本指导书旨在：识别HCCL 8个研发环节中AI能有效介入的切入点，给出可落地的策略和工具建议，并明确AI不应触碰的边界。

### HCCL与周边算子组的复杂性差异分析

conv算子组在AI辅助研发上已有实战复盘[6]，最终代码约70%以上由AI生成。但HCCL不宜直接套用周边算子组的经验，两者在复杂性结构上存在根本差异。

表1 HCCL与周边算子组复杂性对比

| 维度         | 周边算子组(ops)                            | HCCL通信库                         | 对AI策略的影响                                      |
| ------------ | ------------------------------------------ | ---------------------------------- | --------------------------------------------------- |
| 代码模式     | 算子间高度相似（tiling/kernel/host三件套） | 13种通信算法逻辑各自独特           | ops适合模板仿写（实测最大提效点），HCCL仿写空间有限 |
| 核心困难     | 精度、NPU硬件约束、向量化                  | 并发正确性、分布式一致性、协议兼容 | HCCL的核心困难更难被AI消除                          |
| 样板代码占比 | 高（CMake/host/tiling模板）                | 低（逻辑代码占比高）               | HCCL中AI可消除的偶然复杂性比例更小                  |
| 代码规模特征 | 单算子规模小（数百行kernel）               | 单文件规模大（God Object达9152行） | HCCL的AI辅助受上下文窗口限制更为严重                |
| 缺陷可审查性 | 95%（模式化缺陷为主）                      | 85%（并发类缺陷隐蔽）              | HCCL需补充并发专项审查，AI覆盖率上限更低            |
| 测试特点     | 单卡UT/精度对比                            | 多卡集合通信（需物理设备）         | HCCL测试环境搭建的自动化价值更高                    |

核心差异：周边算子组的复杂性中约60%属于偶然复杂性（模板化、重复性工作），AI可消除其中大部分；HCCL各环节偶然复杂性占比从10%到70%不等，按工作量加权后约30%，其中编码和检视环节偶然占比仅35%和15%，AI无法触及并发控制、分布式一致性和协议设计等核心本质复杂性。

### 理论框架：本质复杂性与偶然复杂性

Brooks在"No Silver Bullet"[1]中将软件困难划分为两类：

- 本质复杂性(essence): 问题域固有的复杂度，无法通过工具消除。对应HCCL中的并发控制、通信算法设计、OpCode协议兼容等。
- 偶然复杂性(accident): 当前工具和流程引入的额外摩擦。对应CMake脚本维护、测试脚本编写、日志格式化等。

AI作为工具，主要消除的是偶然复杂性。软件工程历史上三次最大的生产力提升（高级语言、分时系统、统一编程环境）同样如此[1]。AI并非银弹，而是又一次对偶然复杂性的消除。

表2 HCCL各环节偶然复杂性占比

| 环节     | 偶然占比 | AI提效空间 |
| -------- | -------- | ---------- |
| 构建     | ~70%     | 大         |
| 测试     | ~40%     | 中         |
| 编码     | ~35%     | 中         |
| 上板验证 | ~30%     | 小-中      |
| 代码检视 | ~15%     | 小-中      |
| 需求分析 | ~10%     | 小         |
| 详细设计 | ~10%     | 小         |
| 问题定位 | ~10%     | 小         |

注：表2按偶然复杂性占比（即理论提效空间）降序排列。实际ROI还取决于具体策略的投入成本和已有工具基础，详见3.1节。

### 总体策略：精准投放

基于上述分析，HCCL的AI辅助策略应采取精准投放模式——集中资源到偶然复杂性密集的环节，而非全面铺开。

- 综合提效预估：15-20%（vs周边算子组的40-50%）。上限受制于HCCL偶然复杂性占比（各环节加权约30%），详见3.3节计算过程。
- 最高ROI环节：构建（CMake/CI自动化） > 测试（脚本生成/环境配置）。
- 最大风险环节：编码中的并发逻辑生成（AI不擅长并发推理，且该领域恰为HCCL的P0 缺陷域）。

表3 AI自治度分级(Risk Tiering)[2]

| 级别                      | 自治度                     | HCCL适用场景                                |
| ------------------------- | -------------------------- | ------------------------------------------- |
| Tier 1（全自动）          | AI独立执行，人工可选review | 格式修正、CMake检查、测试脚本生成、文档更新 |
| Tier 2（AI执行+自动验证） | AI执行后须通过编译/测试    | 骨架代码生成、构建错误修复、UT框架代码      |
| Tier 3（AI提议+人工审批） | AI给出建议，人工决策       | 影响分析、设计方案探索、并发代码标注        |
| Tier 4（人工主导）        | 人工执行，AI仅提供信息     | 并发逻辑编写、通信协议设计、God Object拆分  |

#### 跨环节通用约束(C1-C8)

以下8条约束适用于所有环节，是逐环节约束的基础。C1-C6为理论约束（源自Brooks框架和HCCL实证数据），C7-C8为流程约束（源自团队流程讨论）。

核心理念：在本质复杂性占70%的系统中，"知道AI不该做什么"比"知道AI能做什么"更关键。AI在HCCL中的最大风险不是"没用好"，而是"用错了地方"。

C1 偶然复杂性占比低于20%的环节，AI仅限信息整理，不参与决策
- 适用：需求(10%)、设计(10%)、问题定位(10%)
- 违反信号：AI的输出被当作技术判断依据而非参考资料

C2 凡涉及正确性判断，必须有确定性验证手段；AI概率性意见不可作为最终判据
- 验证手段：编译器、TSan、ASan、测试
- 违反信号："AI审查通过了所以并发逻辑没问题"

C3 变更爆炸半径越大，AI自治度越低
- 跨层/跨版本/跨设备变更 → Tier 3-4
- 局部格式/文档变更 → Tier 1

C4 AI遗漏的问题必须持久化——可规则化的入规则库，不可规则化的入checklist
- 违反信号：同类问题反复被AI遗漏但无人更新规则

C5 向AI提供上下文：200-500行精准相关代码，不喂整个文件
- God Object（9152行）禁止整体输入
- 违反信号：把整个communicator文件粘贴给AI

C6 每次AI生成代码不超过50行；生成后未经编译验证的代码视为负债
- 违反信号：一次性让AI生成整个函数实现（数百行）

C7 环节衔接：每个环节的产出必须包含AI可消费的格式，为下一环节的AI辅助提供输入
- 需求 →（md+PlantUML+接口定义+参考资料）→ 详细设计
- 详细设计 →（md+PlantUML源码+正常/异常分支100%覆盖）→ 编码
- 编码 →（代码+UT+AI检视报告）→ 代码检视
- 违反信号：上游环节只出了doc/docx或纯截图，下游环节AI无法消费

C8 AI无合入权限——不给予AI +1/+2权限；AI的所有输出必须经人工确认后才产生流程效力
- 违反信号：AI检视通过即视为检视通过

## 逐环节AI辅助策略

### 需求分析（偶然10% /本质90%）

- 责任方：SE
- 输入：业界论文、技术趋势、竞品分析、客户需求
- 输出：需求分析文档（背景、目标、达成路径）

需求分析几乎全是本质复杂性——MPI语义适配、硬件拓扑约束理解、性能指标推导、NCCL/RCCL竞品深度分析，均需系统级架构知识和领域专家判断。AI在该环节仅承担信息整理辅助角色，技术判断与决策权归属领域专家。

与上游的衔接：无（研发链起点）。
本环节产出对下游的约束：需求文档是详细设计环节AI辅助的输入源。

前置条件：
- 无特殊前置。需求分析是研发链起点，AI可直接参与信息整理工作。

红线：
1. AI不得参与技术判断——MPI语义适配、拓扑约束理解、性能指标推导、竞品技术路线选择，均由SE完成
2. AI输出的竞品对比/论文摘要仅作参考材料，不得作为需求决策依据
3. 需求文档中的并发安全不变量（如"不得死锁"）必须由人工定义——AI不理解HCCL分布式通信的全局约束

产出格式约束(C7):
1. 需求分析文档必须有一份md格式
2. 需求分析文档里面的图使用PlantUML绘制
3. 需求分析文档里面包含接口定义
4. （建议）需求分析文档里面包含参考的资料，用于辅助后续详细设计文档生成

判断标准：AI在这个环节做的事，换成一个不懂通信的实习生+搜索引擎也能做 → 用对了；如果需要领域判断才能完成 → 越界了。

#### 策略A-1：论文/竞品分析辅助(Tier 1)

AI辅助阅读和结构化竞品论文（NCCL论文、通信优化论文），输出关键技术点对比表。

- 适用场景：SE需快速了解竞品在算法选择、拓扑适配、性能优化上的做法。
- 约束：AI提供信息整理，技术判断和需求决策由SE完成。
- HCCL定制：对比维度应包括通信算法（Ring/RHD/NHR等）、拓扑适配(HCCS/PCIe/RoCE)、算子覆盖度。

#### 策略A-2：需求结构化模板(Tier 3)

采用EARS格式(Event-Action-Response-State)将通信需求结构化，前置并发安全不变量为可验证约束。

示例："When [rank count > 8 and topology = mesh], the system shall [select NHR algorithm] within [100ms setup time]" → 可直接映射为测试用例。

- HCCL定制：将并发安全不变量（如"任何通信算子在任意拓扑下不得死锁"）作为需求文档的必填约束，而非留到设计阶段。此为周边算子组无需但HCCL必须执行的步骤。

度量指标：
- AI辅助竞品分析的资料整理效率提升（人工整理时间对比）
- 需求文档中结构化约束的覆盖率

### 详细设计（偶然10% /本质90%）

- 责任方：MDE
- 输入：需求分析文档
- 输出：详细设计文档

设计工作几乎全是本质复杂性——通信算法设计（13种算子各自独特）、并发模型设计（线程模型选择、锁粒度决策、内存屏障位置）、状态机设计（executor生命周期）、协议设计（OpCode兼容性）。周边算子组的设计文档以"tiling策略+kernel实现方案"为核心，相对模板化，AI辅助准确率目标可达80%; HCCL设计文档每份独特性高，AI辅助准确率预估50-60%。

vibe-design skill适配度评估：0%（当前无HCCL专项支持）。现有通用模板(generic/srs-sd)未涵盖通信协议设计、并发设计规格、状态机设计、性能设计等HCCL特有章节，需新增HCCL专项设计模板。

与上游的衔接：消费需求分析文档（md+PlantUML+接口定义+参考资料）。上游产出不符合格式要求 → AI辅助设计效果打折。
本环节产出对下游的约束：详细设计文档是编码环节AI辅助的核心输入。

前置条件：
1. 需求分析文档已以md格式交付且包含接口定义——否则AI生成设计骨架缺乏输入
2. 需求文档中的并发安全不变量已明确——否则AI生成的设计骨架缺乏判断基准

红线：
1. AI生成的设计方案不得直接采纳——仅作Disposable Bets（思考起点）
2. 每个AI方案须回答三个必答题： (1)并发安全如何保证？ (2)与现有OpCode兼容吗？ (3)God Object是否需要改动？回答不清晰 → 方案仅作brainstorm素材
3. 并发设计规格（锁粒度、同步原语、屏障位置）不由AI决定
4. AI生成的时序图中的同步点必须人工校验——AI可能遗漏关键同步点

产出格式约束(C7):
1. MDE输出的设计文档必有一份md格式
2. MDE输出的详细设计文档必须有PlantUML的源码，便于下一阶段开发人员用于分析和编码
3. STC评审的时候必须严格把关正常和异常分支覆盖100%——这些分支是编码阶段AI生成UT和代码的判断基准；分支覆盖不全 → 编码阶段AI生成的代码必然有遗漏

判断标准：AI在设计环节只做"搬砖"（文档骨架、图表绘制、格式转换），不做"决策"（算法选择、并发模型、协议设计）。

#### 策略D-1：设计文档结构化辅助(Tier 2)

AI根据设计讨论纪要生成文档骨架（章节结构+占位描述），人工填充核心设计决策。

HCCL定制——模板应包含以下特有章节：
- 并发设计规格：锁粒度、同步原语选择、顺序约束、屏障位置
- 通信协议兼容性：OpCode版本策略、新旧节点互操作方案
- 状态转换图：executor生命周期(sole/parallel/concurrent/sequence)
- 性能设计：内存对齐、cache策略、pipeline配置

#### 策略D-2：图表自动生成(Tier 1)

AI根据文本描述生成PlantUML图（时序图、状态机图、组件图）。

- 适用：将设计讨论成果（白板照片/文字描述）转化为规范化PlantUML。
- 约束：并发时序图需人工校验（AI可能遗漏关键同步点）。

#### 策略D-3: Disposable Bets（一次性方案探索）[2](Tier 3)

针对设计方案选择，令AI以不同角色（并发安全审计员/协议兼容性审查员/性能工程师）分别产出2-3个独立设计方案，设计者从中选择或组合。

- HCCL定制：每个AI方案须回答三个必答题——(1)并发安全如何保证？ (2)与现有OpCode兼容吗？ (3)God Object是否需要改动？
- 约束：AI方案仅作为思考起点，不直接采纳。HCCL设计的本质复杂性需要人类判断。

文档格式要求（从周边算子组复盘迁移[6]）：
- 设计文档使用Markdown格式（doc/docx格式AI理解效果差）
- 图表使用文本化方式(PlantUML/Mermaid)，禁止纯截图
- 文档结构按模块化组织，每个模块对应一个独立的代码生成任务
- 精度标准/性能基线/测试方案须在设计阶段明确（周边算子组教训：未左移导致额外10天重构）

度量指标：
- 设计文档编写时间变化（预期图表绘制时间-70%，文档结构化时间-30%）
- AI辅助各章节的准确率（预期：格式/结构类>80%，并发/协议设计类~50%）

### 编码（偶然35% /本质65%）

- 责任方：开发人员 + IDE
- 输入：详细设计文档
- 输出：功能代码 + UT + LLT

编码是本质复杂性和偶然复杂性并存的环节。偶然部分(35%)包括新算子的目录结构骨架、设备类型分支代码、API注册/导出、错误处理样板、日志打印模板；本质部分(65%)包括并发控制逻辑、通信算法实现、OpCode版本协商、状态机转换逻辑。

AI代码占比预估：30-40%（vs周边算子组70%[6]）。差距源于HCCL本质复杂性占比更高，可由AI生成的偶然代码比例更小。

表4 AI编码禁区及缺陷证据[5]

| 禁区           | 原因                               | 缺陷证据                                        |
| -------------- | ---------------------------------- | ----------------------------------------------- |
| 并发控制逻辑   | AI并发推理能力弱；状态空间指数增长 | HCCL并发缺陷10条/11.9%（全P0），HCOMM 11条/6.8% |
| 通信算法实现   | 13种算法各自独特，非模板可仿写     | 算法正确性缺陷21条/25.0%（HCCL最大缺陷类）      |
| OpCode版本协商 | 协议兼容性属P0级本质约束           | 配置与兼容性缺陷13条/15.5%                      |
| 状态机转换逻辑 | 需对executor生命周期的深度理解     | aicpu_communicator.cc中192行状态机while循环     |

与上游的衔接：消费详细设计文档（md+PlantUML源码+正常/异常分支全覆盖）。设计文档是AI编码的"方案说明书"——没有它AI只会更快地生成错误代码。
本环节产出对下游的约束：代码+UT+AI检视报告是代码检视环节的输入。

前置条件：
1. 详细设计文档已完成并review通过（含STC评审的正常/异常分支100%覆盖）——不跳过设计直接让AI编码
2. 如涉及God Object（3个热点文件），已按功能分区准备好200-500行精准上下文+接口签名；禁止整体输入
3. 错误处理模式库已建立(C-4)——AI引用标准模式，不自行创造新模式
4. 远程编译环境可用——每次生成后须即时编译验证

红线：
1. 四个AI编码禁区（硬性）：并发控制逻辑/通信算法实现/ OpCode版本协商/状态机转换逻辑——AI不得生成这四类代码的实现（可生成接口签名和注释占位）
2. 每次AI生成不超过50行；每次生成后须：编译通过 + 相关UT通过 + 人工review并发相关代码
3. 并发代码变更须附带TSan验证结果(Beyond Done)——无TSan证据的并发修改不得合入
4. AI不得自行创造新的错误处理模式——新模式需经code review确认后入库
5. AI不得判断哪些错误需要重试、哪些需要立即失败——此属通信库业务语义决策

TDD约束：
1. 遵循TDD先行原则：先让AI写UT，UT一旦人工确认通过，后面开发过程中禁止AI再修改UT目录——UT是正确性的锚点，不能随着实现代码一起漂移
2. AI生成的UT必须人工检视确认——AI可能遗漏边界条件或生成形式上通过但实质上不验证任何东西的UT
3. AI生成的Checker必须人工检视确认

PR前约束：
4. 开发者在推送PR前，必须使用AI进行检视，并确认AI的检视意见
5. 开发人员在提交PR前除了AI检视，必须要在本地进行人工检视——AI检视不替代人工检视，两者都是必要条件

判断标准：AI可以写的代码 = 换一个不懂并发的程序员照着模板也能写出来的代码。需要理解线程交互、分布式一致性、协议演进的代码 = AI禁区。

#### 策略C-1：算子骨架生成器(Tier 2)

输入新算子名称+通信语义描述+参考算子名称，输出ops/目录下的algo/executor/selector/template四件套骨架。

- prompt模板："参考HCCL中[AllReduce]算子的目录结构，为新算子[X]生成骨架代码。只生成文件结构、类声明、接口签名，不实现通信逻辑和并发控制。"
- 约束：仅生成结构和接口，不生成并发控制和通信算法实现。生成后须通过编译验证。

#### 策略C-2：小粒度迭代编码规范(Tier 3)

从周边算子组复盘迁移并加强：
- 每次AI生成不超过50行代码
- 每次生成后须： (1)编译通过(2)相关UT通过(3)人工review并发相关代码
- 并发代码变更须附带TSan验证结果(Beyond Done[2])

周边算子组教训[6]直接适用："一次性生成tiling+kernel全部代码，冗余多、幻觉多，导致大量返工"。HCCL中返工风险更高——周边算子组返工主因是精度问题（可测试验证），HCCL返工主因可能是并发问题（难以测试验证）。

#### 策略C-3: God Object上下文管理(Tier 3)

三个热点文件（hccl_communicator_host.cc 9152行/aicpu_communicator.cc 5550行/communicator_impl.cc 3789行）占全部缺陷的36.9%[5]。不应将整个文件输入至AI模型(Context Paradox[2])。

管理策略：
- 按功能分区，每次仅提供相关分区（200-500行）+接口签名
- proximity优先[4]：以同算子的algo/executor/selector/template作为上下文
- 渐进式摘要：对God Object的每个方法生成一句话摘要，构建"方法索引"供AI参考
- 长期方案：God Object拆分（见4.1节）

#### 策略C-4：错误处理模式库(Tier 1)

沉淀HCCL标准错误处理模式（参数校验/返回值检查/日志打印）为code snippet库，AI生成代码时自动引用。

HCCL定制——模式库应包含以下特有pattern:
- ACL接口返回值检查模式
- 通信超时处理模式
- 资源初始化失败的回滚模式（RES-01~05规则[5]）
- 日志tag匹配规则（LOG-01~04规则[5]：格式串参数对应、tag匹配、快速路径避免日志洪泛）

约束：AI使用已沉淀的标准模式，不自行创造新的错误处理模式（新模式需经code review确认后入库）。不由AI决定哪些错误需要重试、哪些需要立即失败——此属通信库的业务语义决策。

#### 策略C-5：远程开发模式优化(Tier 1)

本地IDE(Claude Code / Cursor) + 远端编译验证：
- AI在本地生成代码，通过rsync/ssh同步到远端编译
- 编译结果自动反馈回IDE，形成闭环
- 关键：保持AI工具在本地运行（网络延迟低），仅编译验证走远端

经验迁移[6]："本地承担'思考和写代码'的角色，远端机器承担'编译和跑性能'的角色"。HCCL编译依赖链比周边算子组更深(ACL→HCCL→HCOMM→Runtime→Driver)，且需交叉编译支持多设备类型，编译反馈周期更长，减少不必要的编译往返尤为重要。

度量指标：
- 功能代码中AI生成占比（预期30-40%），其中样板代码AI占比>80%，核心逻辑AI占比<10%
- 编码阶段因并发问题导致的返工次数
- 每次AI生成后的编译通过率

### 构建（偶然70% /本质30%）

- 责任方：开发者 + CI
- 输入：代码
- 输出：编译产物

构建是偶然复杂性最集中的环节，也是AI投入ROI最高的环节。偶然困难集中在CMake脚本维护（新增源文件遗漏、条件编译分支遗漏、链接依赖缺失）、CI流水线配置、构建错误排查。缺陷数据支撑[5]: hcomm-dev 162条缺陷中24条(14.8%)为构建/编译/链接类（频次最高）；hccl-dev 10条中3条(30%)为构建/链接类。

与上游的衔接：消费编码环节产出的代码。构建环节AI的角色是"守门员"——拦截上游遗漏。

前置条件：
1. B-1（CMake一致性守卫）和B-2（编译器诊断全开）已部署——这两项是ROI最高的投入（拦截约55%构建/编码缺陷），在引入任何LLM辅助构建工具之前必须先落地
2. 构建矩阵文档（设备类型x编译选项x依赖版本）已建立

红线：
1. AI给出的构建修复方案必须通过完整编译验证——不接受"注释掉报错代码"类方案
2. AI不得修改交叉编译目标选择和依赖版本锁定——这些是业务约束
3. 编译器诊断渐进式开启——首次全开可能产生大量warning，需分批修复，不由AI一次性"修复"全部（可能引入新问题）

判断标准：构建环节AI的角色是"守门员"（拦截问题）而非"创造者"（生成新代码）。CMake守卫和编译器诊断比任何LLM方案都便宜且可靠。

#### 策略B-1: CMake一致性守卫(Tier 1)

在CI中添加脚本： `find src/ -name "*.cc" | sort` vs `grep "\.cc" CMakeLists.txt | sort`，差集即遗漏的源文件。

- 预期收益：拦截构建类缺陷的40-50%。
- HCCL定制：HCCL的CMake逻辑比周边算子组更碎片化（周边算子组新增算子=固定CMake模式），一致性检查价值更大。
- 投入：低（一个shell脚本）。

#### 策略B-2：编译器诊断全开(Tier 1)

配置： `-Werror -Wshadow -Wconversion -Wformat -Wsign-compare`

- 预期收益：拦截约15%总缺陷（跨仓验证数据[5]）。
- 投入：低（编译选项修改）。
- 注意：首次开启可能产生大量warning需批量修复，建议渐进式开启。

#### 策略B-3：AI辅助构建错误修复(Tier 2)

开发者遇到编译/链接错误时，将错误日志提供给AI分析。

- prompt模板："HCCL构建报错如下： [错误日志]。项目使用CMake构建，依赖ACL/HCOMM/Runtime。请分析根因并给出修复方案。"
- 约束：AI给出的修复须通过完整编译验证，不接受"注释掉报错代码"类方案。
- 参考数据：CI repair成功率17%（GPT-3.5, 170K行上下文）[4]，但HCCL构建上下文远小于170K行，预期成功率更高。

#### 策略B-4：构建矩阵文档化(Tier 1)

利用AI辅助生成和维护构建矩阵文档（设备类型x编译选项x依赖版本）。

- HCCL定制：HCCL需同时支持多种NPU型号(Ascend 910/310/710)和主机架构，编译矩阵比周边算子组复杂。文档化可减少因环境问题导致的构建失败。

B-1与B-2 组合效果：B-1 预期拦截40-50%构建类缺陷（CMake遗漏/链接依赖缺失），B-2 通过编译器诊断额外拦截约15%跨类型编码缺陷（变量遮蔽/类型转换/格式串错误），投入极低。此为HCCL全部AI辅助策略中ROI最高的组合。

度量指标：
- CI流水线中CMake一致性检查拦截的缺陷数
- 编译器warning转error后拦截的缺陷数
- AI辅助构建错误修复的成功率

### 代码检视（偶然15% /本质85%）

- 责任方：开发人员（多人）
- 输入：代码
- 输出：修正后代码

代码检视是本质复杂性最集中的环节之一。85%的复杂性来自需reviewer深度理解的领域：并发逻辑审查（内存屏障位置、锁粒度、Record/Wait顺序）、跨层影响分析(executor→selector→API)、协议兼容性验证（OpCode版本）、缓存一致性判断（cache key维度完整性）。

vibe-review skill适配度：90%。80条HCCL项目级规则已内置[5]，覆盖算法正确性(ALG-01~08)、并发(CON-01~09)、资源生命周期(RES-01~05)、日志(LOG-01~04)等。10%缺口在误报排除（HCCL特有上下文）和跨层影响分析。

HCCL可审查性：85%（vs周边算子组95%）[5]。10%差距来自并发类缺陷的隐蔽性——竞态条件需特定线程调度才暴露，静态审查难以覆盖。提升路径(85%→90%): 不依赖更多AI规则（本质复杂性无法规则化），而依赖TSan/ASan覆盖率从0→80%（消除5%的并发类不可审查缺陷）。

与上游的衔接：消费编码环节产出的代码+UT+AI检视报告。编码环节已要求PR前做AI检视+本地人工检视，代码检视环节是第二道防线。

前置条件：
1. vibe-review已配置HCCL项目级规则（80条）——否则Layer 1自动检视无效
2. 对P0 变更（涉及并发控制）：TSan/ASan基础设施可用——确定性证据 > 概率性意见

红线：
1. AI检视结论不得替代TSan/ASan验证——AI说"并发安全"不算数，TSan说"无竞态"才算数
2. AI不得做最终的并发逻辑正确性判断——Layer 3（并发/跨层/协议）必须由人工完成
3. AI标注的可疑并发问题，即使人工认为"没问题"，也须附加理由记录——防止Eagerness Paradox

责任链约束：
1. AI提出的检视结果必须由开发人员进行确认；最终committer检视或+1的时候必须确认开发人员的反馈结果
2. committer必须对最后一个节点生成的AI检视报告负责——人工确认最后存留的AI检视意见都是误报或已知遗留
3. 开发人员/committer发现AI误报的问题，必须反馈@fanshanshan，用于AI经验积累（Learning Paradox的具体闭环机制）
4. 不给予AI +1/+2权限——AI无合入决策权(C8)

判断标准：AI检视的成功标准不是"发现了多少bug"，而是"释放了reviewer多少注意力给Layer 3"。如果reviewer仍在花时间看格式/规范问题 → Layer 1未达标。

#### 策略R-1：分层检视流程

- Layer 1（全自动，Tier 1）：vibe-review执行编码规范+已知缺陷模式检查。预期节省reviewer 30-40%时间。
- Layer 2（AI辅助+人工确认，Tier 3）：AI标注可疑的并发/缓存/兼容性问题，人工判断。
- Layer 3（人工主导，Tier 4）：并发逻辑正确性、跨层影响分析。

运作流程：
1. 代码提交后，AI自动执行Layer 1检视，输出规范类问题清单。
2. 并行启动Layer 2（AI标注可疑点）与人工检视。
3. 人工聚焦Layer 3的本质问题，不再花时间在已被AI覆盖的格式/规范问题上。
4. 修复所有AI和人工意见后关闭全部检视意见。

#### 策略R-2: Delta-first Review(Tier 3)

AI先分析本次变更影响哪些通信路径，生成影响摘要。reviewer从影响摘要入手而非逐行阅读diff。

- prompt模板："分析以下HCCL代码变更的影响范围。列出： (1)直接修改的函数； (2)调用链上游； (3)调用链下游； (4)是否涉及并发控制、缓存、协议兼容性。"
- HCCL定制：缓解Tunnel Vision Paradox[2]——HCCL的局部修改可能破坏跨层通信语义（修改executor层→影响selector策略→影响API语义）。

#### 策略R-3: N-version并发审查（P0 变更专用）(Tier 3)

对涉及并发控制的P0级变更，令2个独立AI（不同prompt/不同上下文）分别审查，结果交叉比对，不一致处标注为高风险。配合TSan/ASan验证：AI审查结论+工具验证证据方可通过。

- HCCL定制：并发安全需要deterministic evidence（TSan输出），不能仅靠probabilistic review（AI意见）[2]。此为周边算子组无需但HCCL必须执行的步骤。

#### 策略R-4：审查规则持续演化(Tier 3)

- 每次人工review发现AI遗漏的缺陷，分析是否可规则化。
- 可规则化者，更新vibe-review的HCCL规则库。
- 不可规则化者，记录为"需人工关注"的checklist项。

此策略旨在缓解Learning Paradox[2]: 将审查经验编码为persistent rules，不依赖AI的"记忆"。

度量指标：
- AI检视时长vs人工检视时长
- review准确率（目标>=80%）
- AI与人工各自更擅长发现的典型问题类别
- Layer 1自动检视拦截的问题数

### 开发者测试（偶然40% /本质60%）

- 责任方：开发者 + 测试框架
- 输入：功能代码
- 输出：UT + LLT(checker) + 测试报告

测试环节本质复杂性来自分布式通信的不可约简性——多卡/多机集合通信测试需物理设备（环境获取本身是瓶颈）、测试用例设计需理解通信算法语义、并发场景确定性复现困难（LLM故障复现率0%[3]）。偶然部分(40%)集中在测试脚本编写（hccl_test参数组合）、环境配置脚本（SSH免密/rank_table/环境变量）、测试框架代码（setUp/tearDown/mock/断言）。

与上游的衔接：消费编码环节产出的代码。注意：在TDD模式下，UT实际上在编码阶段已由AI先行生成并经人工确认冻结，本环节侧重集成测试/Checker/性能测试。

前置条件：
1. 性能基线已在设计阶段确定（带宽/延迟目标）——否则T-4（测试结果自动分析）无基线可对比
2. 测试环境（多卡/多机）可获取——无物理设备则AI生成的测试脚本无法验证
3. 测试规范要求（功能、性能、长稳、整网）已明确——AI生成的测试用例必须基于规范要求，不得自行发明测试类别

红线：
1. 必须基于测试的规范要求（功能、性能、长稳、整网）来生成测试用例——AI生成的测试用例须覆盖规范定义的全部测试类别，不得遗漏
2. AI不得生成并发测试的时序断言——并发时序属本质复杂性
3. AI不得判断通信结果正确性（如AllReduce结果是否等于各rank输入之和）——通信语义验证需领域知识
4. AI生成的UT桩代码(T-3)必须通过编译验证——LLM对私有API编译率预估<30%
5. AI生成的测试脚本中的参数组合需人工确认覆盖度——AI可能遗漏边界条件（如消息大小为0、卡数为1等）

判断标准：AI在测试环节做"体力活"（参数排列组合、环境配置模板、日志解析脚本），不做"脑力活"（验证逻辑设计、正确性断言、并发场景构造）。

#### 策略T-1: hccl_test矩阵生成器(Tier 1)

输入算子列表+参数范围（卡数、数据类型、消息大小），输出完整的测试命令矩阵+执行脚本。

- prompt模板："为以下HCCL算子生成测试矩阵： [AllReduce, AllGather, ReduceScatter]。参数范围：卡数[2,4，8]，数据类型[fp16,fp32]，消息大小[1KB,1MB,1GB]。输出bash脚本，每个组合一行hccl_test命令。"
- 预期收益：测试脚本编写时间-80%。
- 经验对照[6]：PTA测试脚本生成是周边算子组"分钟级完成（人工需数小时）"的最大提效点之一，可直接复用。

#### 策略T-2：多机环境配置模板(Tier 1)

AI生成rank_table.json模板 + 环境变量设置脚本 + SSH免密配置checklist。覆盖常见拓扑：单机多卡、双机、四机全互联。

- HCCL定制：周边算子组上板为单卡验证，HCCL需要多卡/多机协同，环境配置复杂度高出一个数量级（网络拓扑、HCCL_SOCKET_IFNAME、RDMA参数）。此为HCCL中AI辅助测试的独特价值点。

#### 策略T-3：UT桩代码生成（受限使用）(Tier 2)

AI生成测试框架代码（setUp/tearDown、mock对象），人工填充通信逻辑验证。

约束：
- 不由AI生成并发测试的时序断言
- 不由AI判断通信结果正确性（如AllReduce结果是否等于各rank输入之和）
- 私有API编译率预估<30%（LLM整体编译率58%[3]，私有API更低）

HCCL定制：周边算子组UT为单卡+确定性输入输出，HCCL UT需模拟多rank交互，AI生成难度更高。

#### 策略T-4：测试结果自动分析(Tier 1)

AI编写脚本解析hccl_test输出，提取带宽/延迟数据，与历史基线对比，标注性能退化点。

- 性能基线左移（从周边算子组教训迁移[6]）：在设计阶段确定带宽/延迟目标，编码阶段引入性能冒烟测试，不等到后期。
- HCCL定制：HCCL的性能指标（带宽/延迟）与通信拓扑强相关（HCCS vs PCIe vs RoCE性能差异可达10倍），AI可自动关联拓扑配置与性能数据，发现拓扑特异性退化。周边算子组的性能基线与硬件拓扑无关。

度量指标：
- UT/LLT编码时间变化
- UT覆盖率达标情况
- 测试脚本中AI生成占比
- 性能退化的发现时间点（目标：从ST阶段左移到编码阶段）

### 上板验证（偶然30% /本质70%）

- 责任方：开发者
- 输入：编译产物 + 测试脚本
- 输出：验证结果

上板验证的本质困难在于硬件交互的不可替代性——物理设备获取（多卡/多机为稀缺资源）、NPU行为的不可预测性（温度/功耗/硬件故障）、端到端集成(HCCL+ACL+Runtime+Driver)、性能调优（带宽/延迟与物理拓扑强相关）。偶然部分(30%)集中在环境配置、配置文件准备、测试执行脚本、日志收集与整理。

与上游的衔接：消费编码环节的编译产物+测试环节的测试脚本。

前置条件：
1. 环境检查清单已建立（NPU型号/SDK版本/镜像版本/网络配置/RDMA参数）——V-1的基础
2. 编译环境与运行环境已确认一致——算子组教训：联调环境信息未提前对齐

红线：
1. AI对上板日志的分析仅作初步分类（环境问题vs功能问题）——功能问题的根因分析必须由人工完成
2. AI不得直接修改HCCL运行时参数（HCCL_SOCKET_IFNAME、RDMA配置等）来"修复"上板失败——可能掩盖真正的问题
3. 多机日志的因果推理（rank间消息丢失/乱序）不由AI完成——AI可做时间线重建，人工做因果分析

判断标准：AI在上板验证中做"准备"（环境检查、配置生成）和"整理"（日志收集、时间线对齐），不做"诊断"（根因分析、参数调优）。

#### 策略V-1：环境准备自动化(Tier 1)

AI生成标准化的环境检查脚本：检查NPU型号、driver版本、SDK版本、网络连通性。一键执行，输出"环境就绪"或"以下项目不满足"。

- 经验迁移[6]："联调环境信息未提前对齐，SDK版本不匹配、编译包跨平台不可用"（改进项#9）。HCCL环境复杂度更高（多机网络拓扑、RDMA参数），自动化检查价值更大。

#### 策略V-2：上板日志分析辅助(Tier 2)

上板失败时，AI初步分析日志：区分"环境问题"（配置/版本/网络）与"功能问题"（代码bug）。环境问题AI可直接给出修复建议；功能问题交人工定位。

- HCCL定制：HCCL的上板日志涉及多rank、多机的交叉信息，AI可辅助按rank/timestamp关联日志，构建通信时间线。

联调环境对齐检查清单（从周边算子组改进项迁移[6]）：
- 编译环境（毕昇编译器版本、CMake版本）
- 芯片型号(Ascend 910/310/710)
- SDK版本（cann-toolkit版本）
- 网络配置（HCCL_SOCKET_IFNAME、RDMA参数）
- 镜像版本

度量指标：
- 环境配置时间变化（预期-50%）
- 上板失败中"环境问题"vs"功能问题"的分类准确率

### 问题定位（偶然10% /本质90%）

- 责任方：开发者/问题接口人
- 输入：问题描述 + 日志
- 输出：问题原因 + 修正方案

问题定位是AI最难介入的环节。本质困难包括：Heisenbug（通信超时、死锁的触发条件含不可控因素）、分布式因果推理（多rank间消息丢失/乱序需从多节点日志重建因果链）、性能劣化分析（需理解NPU通信硬件物理特性）、历史知识（根因可能是数月前的设计决策）。Bug localization MAP仅0.39[4]，表明当前LLM在代码理解方面的优势有限。

与上游的衔接：消费上板验证/测试环节发现的问题描述+日志。

前置条件：
1. 问题知识库(P-1)已建立初版——否则AI无法做已知模式匹配
2. 多机日志已收集且时间戳已对齐——否则AI日志分析无输入

红线：
1. AI给出的根因候选（3-5个文件/函数）仅作参考——MAP 0.39意味着约60%概率是错的
2. AI不得做分布式因果推理——多rank间的消息因果关系需人工从通信时间线中判断
3. AI不得给出"问题已定位"的最终结论——只能给出候选范围，人工确认
4. Heisenbug（通信超时、死锁等非确定性问题）不由AI定位——触发条件含不可控因素（硬件状态、网络抖动、线程调度）
5. AI定位无效时，本次case必须补充到问题知识库——闭环

判断标准：AI在问题定位中做"缩小范围"（从430个源文件到3-5个）和"模式匹配"（知识库检索），不做"推理"（因果分析、Heisenbug复现）。

#### 策略P-1：问题知识库(Tier 2)

将历史问题积累为结构化知识库： {错误码，日志特征，根因类别，修复方案}。新问题进来时AI先检索匹配已知模式。

- HCCL定制：通信库的问题模式相对集中——通信超时、死锁、性能退化、协议兼容性错误。知识库的类别体系应围绕这些模式设计。
- 此策略旨在缓解Learning Paradox[2]: 将定位经验持久化为结构化知识，不依赖个人记忆或AI的隐式"学习"。

#### 策略P-2：多机日志分析辅助(Tier 1)

AI编写日志解析工具：收集多机日志 → 时间戳对齐 → 按rank构建通信时间线 → 标注异常事件（超时/错误/重试）。人工从AI整理的时间线中分析根因。

- HCCL定制：此为HCCL独有的需求——周边算子组问题多数可通过DumpTensor+精度对比单卡复现，HCCL问题需要多机日志关联分析。

#### 策略P-3：候选范围缩小(Tier 3)

AI根据错误日志和代码变更历史(git log)，缩小可能的根因范围到3-5个文件/函数。人工从缩小后的范围中精确定位。

- 约束：AI给出的候选范围仅作参考（MAP 0.39[4]表明AI定位准确率有限）。多rank分布式场景下，AI更可能给出误导性的定位结果。

运作流程：
1. AI进行问题定位（知识库匹配 + 日志分析 + 候选缩小） → 人工确认。
2. 若有效，直接按方案修复。
3. 若无效，将本次case补充到问题知识库（闭环）。

度量指标：
- 问题定位时间变化
- AI定位出的问题占比（知识库命中率）
- 知识库条目增长速率

### 环节衔接总览

```
需求分析 ──md+PlantUML+接口定义+参考资料──> 详细设计
                                                  │
详细设计 ──md+PlantUML源码+分支100%覆盖────> 编码
                                                  │
编码 ──────代码+UT(冻结)+AI检视报告────────> 代码检视
    │                                             │
    └──────编译产物+测试脚本──────────────> 上板验证
                                                  │
上板验证/测试 ──问题描述+日志─────────────> 问题定位
                                                  │
问题定位 ──{错误码,日志特征,根因,修复}────> 问题知识库(闭环)
```

关键衔接点：
- 需求 → 设计：接口定义和参考资料是AI辅助生成设计骨架的必要输入
- 设计 → 编码：PlantUML源码让AI理解架构；分支100%覆盖让AI知道要生成哪些分支的代码和UT
- 编码 → 检视：TDD先行的UT（人工确认后冻结）是检视时判断代码正确性的锚点
- 检视 → AI经验积累：误报反馈@fanshanshan → 规则库更新 → 下一轮检视效果提升

约束间的张力：

张力1：C1（偶然<20%仅限信息整理）vs需求阶段仍需AI辅助结构化(A-2)
- 解决：结构化（EARS格式转换）属信息整理，不违反C1；技术判断才违反

张力2：C5（上下文精度200-500行）vs检视时需追踪跨层调用链(R-2)
- 解决：编码用精准分区，检视用影响链追踪——两个环节的上下文策略不同

张力3: C6（50行+即时编译验证）vs HCCL深依赖链导致编译周期长
- 解决：C-5 远程开发模式缓解（本地思考+远端编译）；实际可按"50行生成 → 本地语法检查 → 积攒2-3轮后远端编译"分层验证

张力4：TDD先行（AI先写UT并冻结）vs设计变更导致UT需要修改
- 解决：如设计变更导致UT需修改，须重新走"AI生成UT → 人工确认 → 冻结"流程；不允许静默修改已冻结的UT

## 前置条件与基础设施

以下三项是所有AI辅助策略发挥最大效果的前置条件，优先级高于引入任何新的AI工具。

### God Object拆分

三个热点文件（hccl_communicator_host.cc 9152行/aicpu_communicator.cc 5550行/communicator_impl.cc 3789行）占全部缺陷修复的36.9%[5]。

对AI辅助的阻碍：
- 编码：超出LLM有效上下文处理能力(Bug localization MAP 0.39[4])
- 检视：God Object内的模式匹配更难自动化（Layer 1效果降低）
- 测试：超大类的mock难以生成（UT桩代码质量受限）

拆分优先级高于引入更多AI工具。拆分完成后，所有AI辅助策略的效果均会提升。
实施方式：Risk Tier 4[2](人工主导)，按功能内聚原则逐步拆分，每次拆分须保证编译+测试全部通过。

### TSan/ASan覆盖

当前TSan/ASan覆盖率：0%。

并发安全需要deterministic evidence（工具验证），不能仅靠probabilistic review（AI意见或人工审查）[2]。

目标：TSan/ASan覆盖率从0→80%，预计可消除5%的并发类不可审查缺陷（将HCCL可审查性从85%提升到90%[5]）。

与AI策略的关系：
- 策略C-2（小粒度迭代）中的Beyond Done要求：并发代码变更须附带TSan验证结果
- 策略R-3（N-version并发审查）的必要配合：AI审查结论+TSan证据方可通过
- 构建策略B-2可直接配合ASan：在ASAN构建模式下运行UT

### 文档格式统一

从周边算子组复盘直接迁移[6]: "doc格式+图片内容AI理解效果差；文档上下文过长、模块化不明显"。

要求：
- 设计文档使用Markdown格式
- 图表使用PlantUML/Mermaid文本源码（禁止纯截图）
- 文档结构按模块化组织，每个模块对应一个独立的代码生成任务

此为设计环节(D-1/D-2/D-3)和编码环节（C-1 骨架生成）发挥效果的前提。

### Claude Code CLI

Claude Code是Anthropic的命令行AI编程工具，具备代码库理解、多文件编辑、终端命令执行等能力。它是本指导书中多数AI辅助策略的执行载体——策略C-1（骨架生成）、R-1（分层检视）、D-2（图表生成）等均通过Claude Code落地。

安装：

```bash
# macOS / Linux
npm install -g @anthropic-ai/claude-code

# 验证
claude --version
```

### Skills开发实践

Skills是Claude Code的能力扩展机制——一个SKILL.md文件即可为AI注入领域知识和专项工作流。本指导书中vibe-review（R-1分层检视）和vibe-design（D-1设计文档模板）均以Skill形式实现。

Skill目录结构：

```
<skill-name>/
├── SKILL.md       # 核心：frontmatter + 指令正文（必需）
├── references/    # 参考资料（规则库、大型文档，按需加载）
├── scripts/       # 辅助脚本
└── docs/          # 补充文档
```

SKILL.md编写规范：

| 要素         | 规范                                                    |
| ------------ | ------------------------------------------------------- |
| frontmatter  | `name` + `description`（含触发词）+ 可选`allowed-tools` |
| description  | 列举触发短语，确保用户自然表达即可触发                  |
| 正文语气     | 祈使句为主（"分析…"、"生成…"、"检查…"）                 |
| 篇幅         | 1500-2000字——太短信息不足，太长稀释注意力               |
| 参考资料引用 | 正文中用Read指令按需加载references/下的文件             |

渐进式信息公开(Progressive Disclosure)：

Skill对AI暴露信息分三层，避免一次性灌入过多上下文（C5原则的工具级体现）：

1. metadata：frontmatter中的name + description → Claude Code据此决定是否触发
2. SKILL.md正文：核心流程和约束 → 触发后加载
3. references/ + scripts/：详细规则库和辅助脚本 → 正文中按需Read加载

验证清单：

- [ ] description包含用户常用的触发短语
- [ ] 正文不超过2000字
- [ ] 大型参考资料放在references/下，正文中按需加载
- [ ] 本地测试：在Claude Code中用自然语言触发，验证是否命中

### 贡献方法

每个子模块都是独立仓库，贡献流程遵循标准的Fork → PR模式：

1. Fork目标子仓（而非主仓）
2. 在fork上开发、测试
3. 向子仓提PR
4. 子仓合入后，主仓更新submodule指针

本地调试Skill：

```bash
cd skills/<skill-name>         # 进入子仓目录
git checkout -b feature/xxx    # 创建开发分支
# 修改SKILL.md或references/下的文件
# 在Claude Code中测试触发和执行效果
git add . && git commit        # 提交变更
```

新增子仓：

```bash
git submodule add https://github.com/tsukiyokai/<repo>.git skills/<name>
git commit -m "Add <name> submodule"
```

## 约束与边界

以下约束来自理论框架与HCCL缺陷分析的实证数据。违反这些约束可能导致AI辅助反而增加工作量。

表10 AI辅助约束清单

本表汇总理论约束（源自第5章）和流程约束（源自团队讨论）。完整的跨环节通用约束(C1-C8)及逐环节约束框详见1.4节和第2章各环节。

| 约束项                         | 类型 | 原因                                          | 理论/实证依据                                     |
| ------------------------------ | ---- | --------------------------------------------- | ------------------------------------------------- |
| 不由AI独立编写并发控制逻辑     | 理论 | 并发推理是AI最弱领域，恰为HCCL的P0 缺陷域     | Jagged Intelligence[2] + HCCL并发缺陷11.9%[5]     |
| 不将整个God Object输入AI       | 理论 | 超出LLM有效处理能力，反而降低生成质量         | Context Paradox[2] + 长上下文MAP衰减[4]           |
| 不信任AI的并发正确性判断       | 理论 | AI可能高置信度地给出错误的并发安全判断        | Eagerness Paradox[2] + LLM故障复现率0%[3]         |
| 不跳过设计直接令AI编码         | 理论 | 方案不到位时，AI只会更快地生成错误代码        | 周边算子组核心结论[6] + Vibes-Based Done反模式[2] |
| 不以AI review替代TSan/ASan     | 理论 | deterministic evidence > probabilistic review | AgenticSE Trust框架[2]                            |
| 不期待一个数量级的提效         | 理论 | 核心环节偶然复杂性占比低，综合提效上限15-20%  | Brooks本质/偶然复杂性框架[1]                      |
| 不一次性生成大段代码           | 理论 | 冗余多、幻觉多，返工抵消生成效率              | 周边算子组教训二[6] + HCCL并发返工风险更高        |
| 不由AI判断通信结果正确性       | 理论 | AllReduce/AllGather等语义验证需领域知识       | 周边算子组"方案理解"教训[6]                       |
| 环节产出必须含AI可消费格式(C7) | 流程 | 上游doc/docx或纯截图 → 下游AI无法消费         | 周边算子组教训[6] + 4.3节文档格式统一             |
| AI无合入权限(C8)               | 流程 | AI的所有输出必须经人工确认后才产生流程效力    | AgenticSE Trust框架[2]                            |
| AI检视误报必须反馈闭环         | 流程 | 不反馈 → Learning Paradox无法缓解             | Learning Paradox[2] + R-4 审查规则演化            |
| TDD先行UT冻结后不得修改        | 流程 | UT是正确性的锚点，不能随实现代码漂移          | Beyond Done[2] + 编码环节TDD约束                  |
